Technical Specification v0.1

tastify — API & Package Architecture

A lightweight library for displaying Spotify listening data on developer portfolio sites. React, vanilla JS, and framework-agnostic.

tastify
Spotify listening widgets for developer portfolios.
@tastify/core @tastify/react @tastify/vanilla

Design Principles

01 Tiny footprint

Core <3kb gzip. React ~5kb. Vanilla ~4kb. No one adds weight to a portfolio for a widget.

02 Beautiful defaults, fully overridable

Ships with a refined default theme via CSS custom properties. Headless mode for complete control.

03 Dev Mode aware

Built around Spotify's Feb 2026 restrictions. No batch endpoints, smart request dedup, graceful degradation.

04 Zero config for the common case

One provider, one token endpoint, drop in components. CLI scaffolds the serverless function.

Data Flow

Token refresh lifecycle
┌──────────────┐   GET /api/spotify/token   ┌───────────────────┐
│  Browser      │ ────────────────────────▶ │  Serverless Fn    │
│  (tastify)    │ ◀──────────────────────── │  (your backend)   │
│               │   { access_token, exp }   │                   │
└──────┬───────┘                            └────────┬──────────┘
       │                                             │
       │  GET /me/player/currently-playing           │  POST accounts.spotify.com/api/token
       │  Authorization: Bearer {token}              │  grant_type=refresh_token
       ▼                                             ▼
┌──────────────┐                            ┌───────────────────┐
│  Spotify      │                            │  Spotify Accounts  │
│  Web API      │                            │  Service           │
└──────────────┘                            └───────────────────┘

Refresh strategy:
  1. Client calls getToken() before each API request
  2. getToken() checks if cached token is within expiry window
  3. If expired → hits your /api/spotify/token endpoint
  4. Endpoint uses stored refresh_token → returns new access_token
  5. Client caches token for ~55 minutes (5min safety buffer)
  6. All transparent to the component layer

TastifyClient

@tastify/core — client.ts
interface TastifyConfig {
  /**
   * Static access token — simplest option, expires in 1hr.
   * For persistent sites, use getToken instead.
   */
  token?: string

  /**
   * Async function returning a fresh access token.
   * Called before each request when token is expired.
   * Typically hits your /api/spotify/token endpoint.
   */
  getToken?: () => Promise<string>

  /**
   * Cache TTL in ms per endpoint type.
   * Defaults tuned for portfolio use: now-playing is short,
   * top tracks is long since it changes slowly.
   */
  cacheTTL?: {
    nowPlaying?: number      // default: 30_000  (30s)
    recentlyPlayed?: number  // default: 60_000  (1min)
    topTracks?: number       // default: 300_000 (5min)
    topArtists?: number      // default: 300_000 (5min)
    playbackState?: number   // default: 10_000  (10s)
  }
}

class TastifyClient {
  constructor(config: TastifyConfig)

  // ── Data methods ────────────────────────────────

  /** Currently playing track, or null if idle. */
  getNowPlaying(): Promise<NowPlayingData | null>

  /** User's top tracks for a given time range. */
  getTopTracks(opts?: {
    timeRange?: 'short_term' | 'medium_term' | 'long_term'
    limit?: number  // 1–50, default 10
  }): Promise<TopTracksData>

  /** User's top artists for a given time range. */
  getTopArtists(opts?: {
    timeRange?: 'short_term' | 'medium_term' | 'long_term'
    limit?: number  // 1–50, default 10
  }): Promise<TopArtistsData>

  /** Recently played tracks with timestamps. */
  getRecentlyPlayed(opts?: {
    limit?: number  // 1–50, default 20
  }): Promise<RecentlyPlayedData>

  // ── Polling ─────────────────────────────────────

  /**
   * Polls now-playing at `intervalMs`.
   * Callback fires ONLY when the track changes
   * (deduped by track URI). Returns unsubscribe fn.
   */
  onNowPlayingChange(
    callback: (data: NowPlayingData | null) => void,
    intervalMs?: number  // default: 15_000
  ): () => void

  // ── Lifecycle ───────────────────────────────────

  /** Clears cache, stops all polling. */
  destroy(): void
}

Caching Strategy

@tastify/core — cache.ts
/**
 * Simple in-memory cache with per-key TTL.
 *
 * Design decisions:
 * - No localStorage (SSR-hostile, not available in Workers)
 * - Stale-while-revalidate: returns cached data immediately,
 *   refetches in background if past soft TTL
 * - Hard TTL eviction prevents serving very stale data
 * - Dedup: concurrent requests for same key → one fetch
 */

interface CacheEntry<T> {
  data: T
  fetchedAt: number
  softTTL: number    // serve from cache, trigger bg refresh
  hardTTL: number    // force refetch, don't serve stale
}

/**
 * Request deduplication:
 *
 * If component A and component B both call getTopTracks()
 * in the same render cycle, only one request fires.
 * Both receive the same promise.
 *
 * Critical for Dev Mode — Spotify rate limits are tighter.
 * Dedup prevents wasting quota on duplicate calls.
 */

Design Rationale

font-family: inherit

Widgets should feel native to the host site. Inheriting typography is the single biggest factor in visual cohesion.

BEM classes, not CSS-in-JS

Must work in vanilla HTML with a single stylesheet import. CSS-in-JS requires a runtime. BEM gives targeting precision without tooling.

Transparent bg default

Widgets should feel embedded, not floating. Surface color is opt-in for devs who want a card look.

Spotify green as accent only

Per Spotify's design guidelines, don't replicate their UI. Green is a subtle accent (progress bar, pulse dot), not the theme.

Reference: Vercel Serverless Function

api/spotify/token.ts — generated by npx tastify init
const TOKEN_ENDPOINT = 'https://accounts.spotify.com/api/token'

let cachedToken: string | null = null
let cachedExpiry = 0

export default async function handler(req, res) {
  res.setHeader('Access-Control-Allow-Origin',
    process.env.ALLOWED_ORIGIN || '*')
  res.setHeader('Access-Control-Allow-Methods', 'GET')
  res.setHeader('Cache-Control', 'no-store')

  if (req.method === 'OPTIONS') return res.status(204).end()

  // Serve cached token if still valid (5min buffer)
  const now = Date.now()
  if (cachedToken && cachedExpiry - now > 300_000) {
    return res.json({
      access_token: cachedToken,
      expires_in: Math.floor((cachedExpiry - now) / 1000),
    })
  }

  // Refresh from Spotify
  const params = new URLSearchParams({
    grant_type: 'refresh_token',
    refresh_token: process.env.SPOTIFY_REFRESH_TOKEN!,
    client_id: process.env.SPOTIFY_CLIENT_ID!,
    client_secret: process.env.SPOTIFY_CLIENT_SECRET!,
  })

  const response = await fetch(TOKEN_ENDPOINT, {
    method: 'POST',
    headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
    body: params.toString(),
  })

  if (!response.ok) {
    return res.status(502).json({ error: 'Token refresh failed' })
  }

  const data = await response.json()
  cachedToken = data.access_token
  cachedExpiry = now + data.expires_in * 1000

  // Spotify may rotate the refresh token
  if (data.refresh_token) {
    console.warn(
      '[tastify] Spotify rotated refresh token.',
      'Update SPOTIFY_REFRESH_TOKEN env var.'
    )
  }

  return res.json({
    access_token: data.access_token,
    expires_in: data.expires_in,
  })
}

Security Notes

Never expose the refresh token

The serverless fn keeps client_secret and refresh_token server-side. The browser only sees short-lived access tokens.

CORS lock-down

Set ALLOWED_ORIGIN to your domain in production. Defaults to * for dev but flags this for tightening.

Rate limiting is built-in

Server-side token caching means even high-traffic sites make at most 1 Spotify call per hour. Visitors don't each trigger a refresh.

Token rotation handling

Spotify can rotate refresh tokens. The function logs a warning. A future CLI command could auto-update the env var.

Quickstart — Static HTML (Astro, Hugo, Jekyll)

Full integration in ~10 lines
<!-- 1. Script + styles -->
<script
  src="https://unpkg.com/@tastify/vanilla"
  data-tastify-token-url="/api/spotify/token"
></script>
<link rel="stylesheet" href="https://unpkg.com/@tastify/vanilla/styles.css" />

<!-- 2. Drop in widgets -->
<h2>What I'm listening to</h2>

<div data-tastify="now-playing"></div>

<div
  data-tastify="top-tracks"
  data-header="On Repeat"
  data-time-range="short_term"
  data-limit="5"
></div>

Future Scope

@tastify/web-components

Custom elements (<tastify-now-playing>) via shadow DOM. Same core, zero framework dep. Great for WordPress, Hugo, plain HTML.

@tastify/astro

Astro integration with SSR-rendered stale data + client-side hydration for live updates. Island architecture native.

Apple Music / Last.fm adapters

Same component API, different data source. Core abstracts the provider, making components music-service-agnostic.

Visual theme presets

Curated themes — vinyl, terminal, album-art-dominant, minimal, glassmorphism — toggled via a single CSS class.
